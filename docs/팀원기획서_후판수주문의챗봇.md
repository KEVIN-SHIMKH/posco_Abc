# 후판 수주 문의 응답 챗봇 — 팀원 기획서

- 작성일: 2026-07-21
- 근거 문서: [기획서_후판수주문의챗봇.md](./기획서_후판수주문의챗봇.md)
- 팀 규모: 5인 (역할 기준, 실제로는 겸임 가능 — §4 참고)

---

## 1. 팀 구성 개요

| # | 역할 | 한 줄 정의 | 투입 시점 |
|---|---|---|---|
| 1 | **PM (프로젝트 매니저)** | 범위·일정·의사결정을 관리하고 미결 사항을 해소하는 총괄 창구 | 상시 |
| 2 | **도메인 전문가 (SME)** | 후판 수주 실무 규칙과 실제 데이터를 검증해주는 현업 담당자 | 설계~검증 단계 집중 |
| 3 | **AI/백엔드 엔지니어** | 라우팅·LLM 프롬프트·Flask/SSE 앱을 구현 | 구현 단계 집중 |
| 4 | **데이터 엔지니어** | 엑셀 데이터 로더와 스키마, DB 전환 대비 설계 | 구현 초반 집중 |
| 5 | **QA/검증 담당** | 판정 로직·피드백 루프의 정확성과 예외 케이스를 검증 | 구현 후반~릴리스 전 |

> ※ 애초 검토했던 "소LOT 에이전트 연동 창구" 역할은 제외했습니다 — 실제 연동 대신 안내
> 링크만 노출하는 방식으로 결정되어(§3.5 스텁 범위 축소) 별도 협업 창구가 불필요합니다.

## 2. 역할별 상세

### 2.1 PM (프로젝트 매니저)
- **책임**
  - 프로젝트 범위 관리 — 특히 기획서 §1.2 Non-goals(단중 예측 제외, 최종 조합 설계 확정 금지 등)가 지켜지는지 게이트키핑
  - 우선 구현 순서(기획서 §7)에 따른 일정/마일스톤 관리
  - 미결 사항(기획서 §8)을 SME 등 관련자에게 확인하고 답을 받아 각 역할에 전파
  - 팀원 간 산출물 인터페이스(데이터 스키마, 함수 시그니처 등) 정합성 최종 확인
- **주요 산출물**: 마일스톤 일정표, 미결 사항 트래킹 리스트, 릴리스 체크리스트
- **연계 섹션**: 기획서 §1, §7, §8
- **필요 역량**: 요구사항 정리력, 이해관계자 커뮤니케이션, 우선순위 판단
- **협업 대상**: 전 팀원, SME(도메인 담당자)

### 2.2 도메인 전문가 (SME) — 후판 수주 담당자
- **책임**
  - 사이즈 기준 테이블의 실제 컬럼 구조, 와일드카드 표기 방식, 우선순위 판단 방향 검증
  - 소LOT 판정 threshold와 "주문서 작성 상태"의 실제 기준 필드 제공
  - 집약 기준의 구체적 정의 제공
  - 각 엑셀 데이터(수주실적/여재slab/진행관리)의 실제 샘플·컬럼 목록 제공
  - 판정 로직 결과물에 대한 현업 관점 검수(오탐/누락 여부 확인)
- **주요 산출물**: 실제 엑셀 샘플 파일 또는 컬럼 정의서, 판정 규칙 검증 피드백
- **연계 섹션**: 기획서 §3.3, §3.5, §5, §8-1~4
- **필요 역량**: 후판 수주 실무 지식 (코딩 불필요)
- **협업 대상**: PM(요청 창구), 데이터 엔지니어, AI 엔지니어
- **비고**: 이 프로젝트에서 데이터/규칙 검증의 유일한 병목 — 다른 역할은 대체 인력 투입이
  가능하지만 이 역할은 실제 현업 담당자가 아니면 진행이 막힘

### 2.3 AI/백엔드 엔지니어
- **책임**
  - 문의 유형 판별 로직(규칙 게이트 + LLM tool_use 분류) 구현
  - LLM 프롬프트 설계 — 응답 생성 시 근거자료 외 내용 생성 금지, 참고의견 문구 삽입 로직
  - Flask + SSE 스트리밍 앱, 피드백 입력 UI(경량) 구현
  - 소LOT 스텁 — 조건 판별 후 안내 링크만 노출하는 형태로 구현
- **주요 산출물**: `app.py` (라우팅/프롬프트/Flask 서버), 동작 데모
- **연계 섹션**: 기획서 §3.1, §3.5, §4, §6
- **필요 역량**: Python/Flask, Anthropic SDK(tool_use 포함), 프롬프트 엔지니어링
- **협업 대상**: 데이터 엔지니어(인터페이스), QA, PM

### 2.4 데이터 엔지니어
- **책임**
  - 엑셀 기반 데이터 로더 구현 (`load_size_spec_table` 등, 기획서 §3.2)
  - 사이즈 기준 적합성 판단 로직의 매칭 엔진(와일드카드+우선순위) 구현
  - 추후 DB 전환을 고려한 인터페이스(반환 타입/컬럼 계약) 고정
  - SME로부터 받은 실제 스키마로 기획서 §5의 가정 스키마 교체
- **주요 산출물**: 데이터 접근 계층 함수 세트, 스키마 정의 문서, mock 데이터(SME 데이터 확보 전 임시용)
- **연계 섹션**: 기획서 §3.2, §3.3, §5
- **필요 역량**: pandas, 데이터 모델링
- **협업 대상**: SME, AI 엔지니어, QA

### 2.5 QA/검증 담당
- **책임**
  - 사이즈 판정 로직의 엣지 케이스 검증 (매칭 규칙 없음, 규칙 충돌, 경계값 등)
  - "확인 필요" 폴백이 실제로 임의 추정 없이 동작하는지 검증 (기획서 §1.3 원칙 준수 확인)
  - 정형 데이터 조회 응답에 참고의견 문구가 누락 없이 포함되는지 확인
  - 피드백 저장 기능 정상 동작 확인 (로그 유실 없음, 형식 일관성)
- **주요 산출물**: 테스트 케이스 목록, 결함 리포트
- **연계 섹션**: 기획서 §3.3, §3.6, §1.3
- **필요 역량**: 테스트 설계, 도메인 룰에 대한 기본 이해
- **협업 대상**: AI 엔지니어, 데이터 엔지니어, PM

## 3. 역할 간 협업 흐름

```
SME ──(실제 데이터/규칙 제공)──▶ 데이터 엔지니어 ──(스키마/조회 함수)──▶ AI 엔지니어
                                                                        │
PM ──(범위/우선순위/미결사항 확인)──▶ 전 팀원                              │
                                                                        ▼
                                                              QA ◀── 구현 결과물 검증
                                                                        │
                                                                        ▼
                                                              PM ── 릴리스 판단
```

## 4. 겸임 가능성 (소규모/1인 진행 시)

실제 인원이 제한적일 경우 아래처럼 압축 가능합니다.

| 압축 시나리오 | 겸임 구성 |
|---|---|
| 최소 인원 | PM + AI 엔지니어(데이터 엔지니어 겸) 1인, SME는 외부 자문으로 분리, QA는 AI 엔지니어가 셀프 체크 |
| 권장 최소 | PM 1인, 개발(AI+데이터 엔지니어 겸) 1인, SME(외부 자문), QA는 개발자 겸임 |

다만 **SME 역할만은 겸임 대상에서 제외**를 권장합니다 — 도메인 지식 없이 사이즈 기준/소LOT
threshold를 임의로 가정하면 기획서 §1.3 원칙(검증 안 된 데이터 임의 생성 금지)이 실질적으로
지켜지지 않습니다.
